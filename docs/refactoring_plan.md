# План рефакторинга структуры (Чистота файлов)

Корневая папка проекта не должна быть свалкой (избавляемся от эффекта "разбросанных тюленей"). Любые служебные скрипты, тестовые конфигурации и инструменты для дебага должны находиться строго на своих местах.

## 1. Скрипты и Дебаг (`scripts/`)

**Цель:** Изолировать всю автоматизацию, которая не относится напрямую к коду приложения.

- Скрипты для локального дебага (типа `debug.sh`), сценарии рестарта Docker, скрипты очистки кэшей должны лежать в папке `scripts/`.
- Вызов этих скриптов (если сильно нужно) прокидывается в секцию `scripts` файла `package.json`, чтобы разработчики могли вызывать их как `npm run debug`.

## 2. Тесты и тестовые файлы

**Цель:** Организовать хранение тестов.

- **Unit тесты:** (например, `*.test.ts`) должны лежать **строго рядом с тем кодом, который они тестируют**. Это позволяет быстрее находить тесты к файлам и не плодить искусственные деревья папок `tests/` с дублирующей иерархией.
- **E2E / Интеграционные тесты:** Если тесты проверяют сразу несколько сервисов (например, взаимодействие Gateway и Python Backend), они должны быть вынесены в отдельную папку `e2e_tests/` (или `integration/`), так как они не принадлежат ни к одному из изолированных компонентов.
- **Фикстуры (mock data):** Тестовые документы, картинки (pdf, png) должны лежать в папке `tests/fixtures/`, чтобы не мусорить рядом с кодом компонента.

## 3. Конфигурации (Линтеры, Сборщики)

- Глобальные файлы (например, `.gitignore`, базовый `package.json`, базовый `.dockerignore`, `eslint.config.js` если он на весь воркспейс) могут оставаться в корне.
- Локальные настройки (например, `vite.config.ts`, `tsconfig.json` для реакта) должны лежать непосредственно в папке `web/`. То же касается настроек `Python` (файлы `pyproject.toml`, `.flake8` лежат в папке `ocr/`).

## Итог идеального корня (Root Folder):

В корне должны остаться только:

- Папки подпроектов: `web/` (Фронт), `gateway/` (Шлюз), `ocr/` (Backend), `scripts/` (Bash-скрипты), `docs/` (Документация).
- Глобальные конфигураторы Docker: `docker-compose.yml`, `*.Dockerfile`.
- Мета-файлы пакетов: `package.json`, `package-lock.json`.
- Точка входа для развертывания: один `run.sh` (в состоянии "curl this file to start the whole app").
- Описание проекта: `README.md`.

## 4. План рефакторинга `App.tsx` (Устранение монолита)

**Цель:** Разделить "стену кода" и монолитный старт приложения на логические модули. Изолировать типы, интерфейсы и выделить единую зону навигации.

**Предлагаемая архитектура:**

1. **Единая навигационная область (Шапка + Боковая панель):**
   - Шапка (`Header.tsx`) и боковая панель (`Sidebar.tsx`) объединяются в единую логическую обертку управления — `web/src/ui/layout/NavigationArea.tsx`.
   - Это гарантирует, что все элементы управления (выбор движка, настройки, статусы задач) имеют единый контекст и логику отображения.
   - **Интерфейс-прокладка:** Навигационная область жестко связывается с логикой извлечения текста через отдельный связующий слой интерфейсов (например, `web/src/ui/layout/engine-controls.types.ts`). Это изолирует UI-компоненты от прямого встраивания логики подкапотного OCR-движка.

2. **Интерфейсы и Типы:**
   - Вынос общих структурных интерфейсов приложения (темы, layout) в `web/src/types/app.types.ts`.
   - Интерфейсы состояния извлечения (запущен, отменен, загрузка) выделяются изолированно, связывая навигацию и саму логику пайплайна.

3. **Бизнес-логика (Main Content Zone):**
   - Зона работы с файлом (Drag & Drop, просмотр изображения, canvas) выносится в отдельный компонент `web/src/ui/workspace/OcrWorkspace.tsx`.
   - Четкое разделение рабочей области и навигационной области предотвратит перерендеринг холста при изменении статуса в боковой панели.

4. **Точка входа (`App.tsx` и Context API):**
   - Само тело `App.tsx` максимально "худеет" и становится только "узлом сборки" приложения.
   - В нём остаётся инициализация Провайдера Состояний (`OcrContext`).
   - Массивный хук `use-extraction.ts` инкапсулируется в состояние Context API, предоставляя дочерним компонентам (`NavigationArea` и `OcrWorkspace`) только строгие контракты действий (например: `startExtraction()`, `status`), убирая необходимость прокидывать десятки свойств (`props`) через всё дерево компонентов.
