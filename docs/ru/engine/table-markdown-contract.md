# Markdown-контракт таблиц

Этот контракт описывает, как backend должен записывать OCR-таблицы в Markdown,
если исходная таблица содержит объединённые ячейки.

GitHub Flavored Markdown не поддерживает `rowspan` и `colspan`, поэтому backend
пишет обычную pipe-table с фиктивными placeholder-ячейками. UI читает эти
placeholder-ячейки и визуально восстанавливает объединение. Кнопка
`Копировать всё` копирует исходный Markdown с placeholder-ячейками, а не HTML.

## Маркеры

| Маркер              | Значение                                                  |
| ------------------- | --------------------------------------------------------- |
| `::merge-left::`    | эта ячейка занята ближайшей обычной ячейкой слева         |
| `::merge-up::`      | эта ячейка занята ближайшей обычной ячейкой сверху        |
| `::merge-up-left::` | эта ячейка занята прямоугольным merge-блоком сверху-слева |
| пустая ячейка       | настоящий пропуск данных, не объединение                  |

Backend не должен использовать пустую строку как сигнал объединения. Пустая
ячейка `|  |` означает только отсутствие текста в этой клетке.

## Горизонтальное объединение

Исходная строка с `colspan=4`:

```markdown
| РАЗДЕЛ A / SECTION ALPHA | ::merge-left:: | ::merge-left:: | ::merge-left:: |
```

UI отображает это как одну широкую ячейку, но raw Markdown остаётся пригодным
для копирования и debug-снимков.

## Вертикальное объединение

Исходная первая колонка с `rowspan=3`:

```markdown
| Семестр 1 | Математика |
| ::merge-up:: | Физика |
| ::merge-up:: | Информатика |
```

UI отображает первую ячейку как вертикально объединённую. Остальные колонки
остаются обычными.

## Прямоугольное объединение

Если ячейка занимает несколько строк и несколько колонок:

```markdown
| Блок | ::merge-left:: | Значение |
| ::merge-up:: | ::merge-up-left:: | 10 |
| ::merge-up:: | ::merge-up-left:: | 20 |
```

Правило: в первой строке merge-блока используются `::merge-left::`, а в нижних
строках первая покрытая колонка использует `::merge-up::`, остальные покрытые
колонки используют `::merge-up-left::`.

## Debug sample

В web UI есть dev-only fixture для проверки renderer-а:
`SAMPLE_mixed_ru_en_zh_table_image.pdf`. В dev-сборке этот файл подставляет
ручной Markdown с merge-маркерами, чтобы проверять UI отдельно от OCR-движка.

Production OCR не должен зависеть от имени этого файла. Реальный backend должен
создавать такие маркеры из распознанной геометрии таблицы.
