# Архитектура и Принцип Слоев (Универсальная сборка)

Отказываемся от понятия "монолитный план" и переходим к чистой логике слоев. Код должен работать и на утюге, и в дата-центре, но конфигурации для них кардинально разные.

## 1. Уровни абстракции (Слои)

Система делится на четкие изолированные слои. Их можно собирать как лего в зависимости от того, куда мы деплоим (от расширения в браузере до кластера).

1.  **CLIENT (UI & State):** HTML/JS/CSS (Vite/React). Тупой, но красивый. Ничего не вычисляет, только показывает кнопки, хранит состояние и шлет HTTP-запросы.
    - _Включает WASM:_ WebAssembly (например, локальный tesseract.js) как резервный "очень слабый" оффлайн-движок прямиком в браузере на крайний случай.
2.  **EDGE & TUNNELS (Cloudflare):**
    - _Workers:_ Замена традиционного шлюза для LITE-сборок. Живет на серверах Cloudflare. Прячет API-ключи от Gemini/OpenAI, может делать легкие проверки, но не может крутить нейросети.
    - _Tunnels (cloudflared):_ Тот самый "туннель сквозь них". Позволяет прокинуть твой локальный **GATEWAY** или **ENGINE** наружу без белого IP и возни с роутерами, используя железо Cloudflare как балансировщик и DDoS-защиту.
3.  **GATEWAY / SERVICE (API & Queue):** Приложение на Node.js/Bun (`server.ts`). Тот самый "швейцар". Балансирует нагрузку, держит очередь (чтобы Ollama не сдохла от 10 запросов), занимается CORS и раздает статику, если Nginx нет.
4.  **ENGINE (Heavy Compute):** Питоновский сервис (`ocr/app/main.py`), напичканный PyTorch, EasyOCR и прочими гигабайтными зависимостями. Жрет CPU/GPU. Делает только одно: получает картинку -> отдает текст.
5.  **PROXY (Ingress):** Nginx. Раскидывает трафик, кеширует статику, терминирует SSL.
    - _Зачем он, если есть Bun/Node?_ Потому что Nginx написан на C и использует `sendfile` на уровне ядра ОС. Nginx отдает готовую статику и держит DDoS в сто раз лучше любого энтерпрайзного JS-фреймворка. Он выступает как бронированная дверь, через которую мы делаем `proxy_pass /api/` прямо в сервис Bun/Node, не пуская мусорный трафик на вычисления.

## 2. Формы существования (Целевые сборки)

Исходя из этих слоев, мы собираем проект тремя способами:

### 1. Форма "LITE" (GitHub Pages / Extension / Hyprland) -> [CLIENT] + [EDGE]

Только статика в браузере клиента (расширение, Web, GitHub Pages).

- **Где тут Nginx? (Спойлер: его тут быть не может):** Ты спрашиваешь, почему бы не поднять Nginx без контейнера в LITE-версии для стабильности. Собери свою логику в кулак: куда ты собрался устанавливать системный Nginx в _расширении для Firefox_ или на _бесплатном хостинге GitHub Pages_? У тебя нет операционной системы, нет сервера, нет доступа к портам 80/443. LITE — это тупо набор HTML/JS файлов, живущих в песочнице браузера пользователя.
- **Логика:** В статическом клиенте _физически не может быть централизованного швейцара_. Браузер — это песочница. Две открытые вкладки не знают друг о друге.
- **Приоритет маршрутизации:** Клиент _ВСЕГДА_ сначала стучится напрямую (например, к `window.location.origin` на VPS или к заданному локальному IP `127.0.0.1` на твоей машине) — потому что приоритет отдается мощностям локального сервера с ядрами, за который ты уже заплатил.
- **Если нужна защита или хост умер (Fallback):** Если прямая связь недоступна, запросы улетают на уровень **EDGE** (Cloudflare Worker). Это твой запасной аэродром, который жонглирует API и маршрутами извне.
- **А если стучаться напрямую в Ollama:** Если юзер откроет 10 вкладок и нажмет "Оцифровать", локальная Ollama словит инсульт. У LITE-версии нет серверных мозгов для глобальной очереди. Поэтому fallback через Cloudflare или использование локального Gateway строго необходимы для адекватной работы.

### 2. Форма "STABLE" (Bun / Localhost / Bare-metal VPS) -> [PROXY] + [CLIENT/GATEWAY] + [ENGINE]

- **Логика:** У нас есть голая ОС (Ubuntu/Windows). Мы запускаем `node server.cjs` (или `bun server.ts`). Он становится **GATEWAY**.
- **Тот самый Nginx без контейнера:** Вот здесь, на "железной" VPS без докера, твое желание исполняется. Мы ставим системный Nginx (`apt install nginx`) прямо перед Bun/Node-сервером. Он моментально раздает статику файлов из Vite и выступает как непробиваемый щит от мусорных запросов. А `bun` тихо сидит на порту локалхоста.
- **Очередь:** Вот тут швейцар (Gateway) работает на 100%. Хоть 100 вкладок открой, он выстроит их в очередь и будет отдавать запросы локальной Ollama или локальному питоновскому встроенному парсеру (через `venv`) строго по одному.

### 3. Форма "DOCKER" (Тяжелый VPS) -> [PROXY] + [CLIENT/GATEWAY] + [ENGINE]

Жесткая изоляция 3 контейнеров (`docker-compose.yml`).

- **Почему не 1 монолит?** Node.js весит ~100 МБ. Python с ML-библиотеками (PyTorch) весит 5+ ГБ. При любом изменении строчки в Node.js ты будешь пересобирать этого 6-гигабайтного монстра бесконечно долго. Плюс, если упадет Node.js (из-за кривого JSON), он убьет тяжелый процесс Python, сбрасывая состояние ML-моделей.
- **Архитектура докера:**
  1. `nginx-container` — ловит 80/443 порты. Раздает статику сайта. Пробрасывает API в Gateway.
  2. `gateway-container` — Образ `node:20-slim` (или `node:20-alpine`). _Почему не Bun?_ Bun хорош на твоем локалхосте, чтобы не ждать компиляции TS. Но в Docker-продакшене правит бал индустриальный стандарт стабильности. Собранный Node-slim весит жалкие ~50 MB, потребляет крохи памяти и гарантированно не падает с `Segmentation fault` в экзотических нагрузках.
  3. `ocr-engine-container` — Изолированная банка с Python. Делает тяжелую работу.

## 3. Разрешение конфликтов: Docker vs Venv (Смерть монолитного run.sh)

Зачем допускать одновременное существование `docker` и `venv` в проекте?
Потому что **DOCKER** — это для развертывания на боевом сервере (чистота и повторяемость), а **VENV + BUN/NODE** — для быстрого локального запуска (без оверхеда контейнеризации на слабом железе).

Как мы не сойдем с ума? **Мы убиваем тупой монолитный `run.sh`, который пытается сделать всё сразу.**

- `scripts/run-docker.sh` (или просто `docker-compose up`) — поднимает тяжелую артиллерию. Никаких `venv` на хосте, всё варится внутри контейнеров.
- `scripts/run-local.sh` (STABLE) — запускает локальный Node.js Gateway и дергает локальный Python через `venv`.
- `scripts/build-lite.sh` (LITE) — просто компилирует Vite (HTML/JS) для закидывания в браузерное расширение или на GitHub Pages. Ничего не запускает.

Таким образом `docker` и `venv` существуют в одних исходниках, но вызываются **разными точками входа**, не пересекаясь.

## 4. Как тестировать этот зоопарк (GitHub Actions)

Раз у нас три разные среды обитания, тупой `npm test` нас не спасет. Мой план для пайплайнов (YAML), чтобы покрыть всю эту мешанину без слез:

1. **Линтеры и статика (Fast Fail):** Бегут мгновенно. ESLint для JS/TS, `ruff` и `mypy` для Питона. Убивает дурацкие ошибки до того, как начнется долгая сборка.
2. **Юнит-тесты модулей (Изоляция):**
   - `pytest` для **ENGINE** (проверяем парсеры кусков текста, мокаем нейросети, скармливаем заранее заготовленные картинки-фиксчеры).
   - `vitest` для **GATEWAY** (проверяем логику очередей швейцара и валидацию API-входов).
   - **А что с Bun? (Ответ на твое нытье):** Мы не "оставляем его разбираться сам с собой". Bun и Node — это две разные программы, но они выполняют _один и тот же_ JavaScript-код. Чтобы убедиться, что код стабилен в обоих рантаймах, в GitHub Actions мы используем **Matrix Build**. Один и тот же код Gateway будет тестироваться `vitest`-ом в контейнере `node:20`, а затем параллельно — в контейнере `oven/bun`. Если Bun словит сегфолт или баг на внутренних вызовах (что он любит делать) — пайплайн сразу покажет тебе, какой из рантаймов отвалился.
3. **E2E Тесты (Интеграция всего):** Самый сок. GitHub Actions буквально делает `docker-compose up -d`. Поднимаются все 3 контейнера. Затем запускается Playwright (безголовый браузер Chromium), общается с живым `nginx` и прокликивает UI, имитируя пользователя. Если туннель от фронта через шлюз до питона и обратно не разорвался — сборка зеленая.
