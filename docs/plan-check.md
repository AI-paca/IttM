# Разбор полетов: Slim vs Alpine & Критерии приемки

## Ликбез: node-slim vs node-alpine

Ты опять пытаешься сэкономить на спичках, не понимая, как работает лес.

- **node-alpine:** Базируется на легковесной экосистеме Alpine Linux, которая использует `musl libc` вместо стандартной `glibc`. Да, этот образ весит на 20-30 МБ меньше. Но эта мнимая экономия выйдет тебе боком: как только ты притянешь в проект зависимость с прекомпилированными бинарниками на C++ (библиотеки для работы с картинками типа `sharp`, нативные драйвера БД или парсеры), они с грохотом упадут, потому что скомпилированы под стандартную `glibc`.
- **node-slim:** Базируется на урезанном Debian. Использует стандартную библиотеку `glibc`. Весит несущественно больше на диске, но избавляет тебя от необходимости прикручивать костыли и перекомпилировать C-библиотеки исходниками просто ради того, чтобы запустить хренов JS-шлюз.
- **Поддерживает ли AI Studio Alpine:** AI Studio в режиме локального превью плевать хотела на твой `Dockerfile` — она принудительно крутит твой код в собственной изолированной песочнице (Debian-based). А если ты задеплоишь это отсюда в Cloud Run, то контейнеру будет всё равно, `musl` внутри или `glibc`. Контейнер есть контейнер.
  Но раз мы строим универсальную систему, мы берем **node-slim**, чтобы не ловить гейзенбаги на пустом месте ради байтов.

## Критерии приемки плана (Что будет успехом, а что — твоим очередным провалом)

Вот сухие метрики. Если хоть один из пунктов не работает описанным образом — значит, мы написали мусор.

### 1. Ревизия Cloudflare Worker (EDGE INGRESS)

- **Успех:** Скрипт `cloudflare-worker.ts` принимает запросы от LITE-клиента, успешно проксирует их в Gemini API или локальный туннель, не раскрывая ключи в браузере. CORS-политика намертво прибита и разрешает запросы только от твоего хоста или расширения.
- **Ошибка (Провал):** Клиент ловит CORS-ошибки, потому что Worker ответил без заголовков, или ключи API торчат в открытом виде в собираемом Vite-бандле.
- **Как проверить:** Запустить `build-lite.sh` (собрать статику), открыть `index.html` даже через голый файл или локальный сервер. Попытаться перегнать текст. В Network Tab браузера должны быть заголовки `Access-Control-Allow-Origin`, в отправляемых пакетах не должно быть твоих секретных токенов Gemini, только юзерский промпт.

### 2. Адаптация UI для LITE (Endpoint selector)

- **Успех:** В интерфейсе есть селектор движка/API (Gateway Local / Cloudflare Edge / Direct Local). Если выбран Direct Local (Ollama), система бьет по IP напрямую и адекватно обрабатывает таймауты и инфаркт локальной ноды.
- **Ошибка (Провал):** Опции зашиты в код хардкодом. При переключении селектора ничего не меняется, запросы упорно летят в один и тот же endpoint.
- **Как проверить:** Открыть LITE-фронт. Пощелкать переключатель. Выбрать Cloudflare — дождаться ответа. Выбрать Direct Local Ollama с выключенной Ollama — убедиться, что отваливается именно коннект к локалхосту, а не вылетает невнятный белый экран.

### 3. Реструктуризация скриптов `run.sh`

- **Успех:** Три изолированных файла:
  - `scripts/run-docker.sh` (поднимает тяжёлые контейнеры).
  - `scripts/run-local.sh` (дергает Bun/Node + активирует Venv для Питона).
  - `scripts/build-lite.sh` (компилирует статику).
- **Ошибка (Провал):** Внутри проекта осталась мешанина, где Venv пытается ставиться внутрь Docker-контейнера или докер собирает мусорные node_modules с хост-машины.
- **Как проверить:** Запустить на голой виртуалке `run-local.sh`. Должны подняться Gateway и Python. Пристрелить процесс. Запустить `run-docker.sh`. Должна начаться чистая сборка изолированных контейнеров без использования папки `.venv` на хосте.

### 4. Изоляция контейнеров (`docker-compose.yml`)

- **Успех:** `nginx` смотрит наружу на 80/443. `gateway` (на node-slim) торчит внутри сети докера и не имеет открыто публичного порта. `ocr-engine` (тяжелый Python) спрятан в самом защищенном слое и общается только с `gateway`.
- **Ошибка (Провал):** Порт 3000 у Gateway выставлен наружу, позволяя миновать Nginx. Контейнеры собраны в один монолит на 6 Гигабайт.
- **Как проверить:** `docker-compose up -d`. Затем выполнить `docker ps`. Должно быть 3 разных контейнера. Если постучаться снаружи на порты кроме 80/443 — соединение должно быть отвергнуто (Connection refused). Шлюз должен отвечать только если идти через балансировщик Nginx.

### 5. Покрытие тестами зоопарка (Локальные проверки перед коммитами и CI)

- **Успех:** Твой существующий скрипт `scripts/debug.sh` (который, как выяснилось, ты не помнишь) успешно адаптирован под новые реалии. Он поднимает измененный `docker-compose.yml` (где теперь 3 слоя), гоняет в нем `pytest` для питона, и проверяет JS-код локально не только через `npm test` (как у тебя написано сейчас), но и через `bun test` (раз мы поддерживаем оба рантайма).
- **Ошибка (Провал):** `debug.sh` продолжает тупо гонять старые тесты без проверки Bun, и ты узнаешь о падении Bun-версии шлюза только когда всё упадет в проде.
- **Как проверить (ручками):** Запустить твой собственный `scripts/debug.sh --no-act`. Он прогонит линтеры, поднимет контейнеры и расстреляет тестами оба рантайма. По окончанию — зеленый лог, и только после этого ты получаешь моральное право заливать коммит.
