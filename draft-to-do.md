# Draft To-Do — что не сделано относительно целевой архитектуры

> Контекст: текущий код считается **целевой (релизной) реализацией**. Этот файл
> фиксирует пробелы между тем, что описано в документации как релизное
> состояние, и тем, что потребовало бы идеальная архитектура. Это не бэклог
> «к релизу», а инвентаризация незакрытых зон.
>
> Деление:
>
> - **to-do** — нужно довести до целевого состояния. Акцент на тестах и их
>   покрытии: баги/уязвимости, которые документ обещает закрытыми, но код/CI их
>   не гарантируют.
> - **not-to-do** — только проверить текущее поведение и задокументировать;
>   намеренно **не** исправлять, чтобы не сломать архитектурные компромиссы.

## to-do — довести до целевого состояния

### Task lifecycle / durability

- **Durable queue и retention отсутствуют.** In-memory task queue
  (`maxWorkers: 1`, `maxQueued: 32`) даёт task ID и cancel, но задачи теряются
  при рестарте gateway. Документация это declares, но целевая архитектура
  «production-grade» требует persistence/резюма после падения.
  - Покрытие: нет теста, что после рестарта процесса очередь корректно
    деградирует (а не молча теряет задачи без сигнала клиенту).
- **Timeout/abort проксирования.** В `routes.test.ts` тестируется только happy
  path. Нет тестов на обрыв связи, таймаут upstream (Python OCR) и abort
  клиентского запроса посреди multipart.
- **Retry-политика для upstream OCR.** Сетевая ошибка gateway↔OCR не
  ретраится и не помечает task как переисполнимый.

### Размер и форма входа (DoS / bombs)

- **Size limit middleware на gateway/Express.** Python ограничивает upload
  (`OCR_MAX_UPLOAD_BYTES`), nginx — `client_max_body_size`, но в самом
  gateway нет раннего отказа на `Content-Length`. Нужен контракт-тест, что
  oversized payload отвергается на gateway до проксирования.
- **Decompression bomb coverage.** `OCR_MAX_DECODED_IMAGE_PIXELS` (`80MP`) —
  это guard, а не tile decoder. Нет fuzz/метрик-тестов на pixel bombs
  (малый encoded размер, гигантское разрешение) и на corrupted buffers,
  которые у senior-ревью вызывают segfault/OOM. Целевое состояние: API
  возвращает `400`, а не падает процессом.
- **Sanitization расширений/имен.** Суффикс берётся из имени пользователя.
  Нужен тест, что произвольные суффиксы (`.tar.gz.sh`, двойные расширения) не
  влияют на диспетчеризацию библиотек.

### Abuse / rate limiting

- **Нет rate limiting на публичных маршрутах.** `/api/extract/text`,
  `/api/convert*`, `/api/tasks` не защищены от abuse (CPU Tesseract/EasyOCR).
  Целевая архитектура предполагает хотя бы per-IP throttle в loopback-режиме.
- **Тест на concurrent flood.** Нет теста, что `maxQueued: 32` корректно
  отвергает 33-ю задачу (429/503) и не приводит к OOM.

### CORS / surface

- **Strict CORS для OCR-сервиса.** `ocr/app/main.py` использует
  `allow_origins=["*"]`. В Docker это безопасно (внутренняя сеть), но при
  bare-metal запуске `127.0.0.1:8000` открыт для любого сайта. Целевое
  состояние: CORS должен зависеть от режима запуска.

### Качество OCR и регрессии

- **Golden master / CER/WER gate в CI.** Тесты качества (`test_ocr_quality.py`)
  есть, но нет автоматического block-on-regression: пулл-реквест, ухудшающий
  распознавание «колбасных» скриншотов, не будет остановлен CI. Нужен
  baseline + порог `character_error_rate` как PR gate.
- **Кросс-движковый паритет.** Browser WASM и Python Tesseract дают разные
  результаты на одном файле. Нет контракта, декларирующего, что WASM — это
  fallback сниженного качества (и теста, фиксирующего допустимую дельту).

### Cleanup / observability

- **Tempfile cleanup при crash.** Нет теста, что PDF-каталоги в `tempfile`
  удаляются при жёстком падении процесса, а не только в `finally`.
- **Логи/метрики task lifecycle.** Нет observability-теста: события
  `queued → running → done/cancelled` должны быть трассируемы для одного task
  ID.

## not-to-do — проверить, не исправлять

> Эти пункты — **осознанные архитектурные компромиссы**. Их нужно периодически
> перепроверять (что поведение осталось заявленным), но не «чинить» ради
> маркетинга, иначе ломается local-first/privacy-first модель.

- **External LLM — Base64 payload на клиенте.** Крупные файлы кодируются в
  Base64 перед отправкой провайдеру. Это намеренно: ключ не уходит в локальный
  OCR, а провайдер принимает base64. Проверять: размер payload и freeze UI
  остаются в заявленных пределах; не двигать обработку в backend.
- **Browser OCR downscale (`2200-4200px`/`4-14MP`).** Downscale может терять
  мелкие цифры/тонкие линии. Это компромисс против OOM вкладки. Проверять:
  лимиты worker-resize соответствуют профилю; не включать полноформатную
  обработку по умолчанию.
- **Горизонтальное масштабирование Tesseract/EasyOCR.** CPU-bound движки не
  масштабируются горизонтально из коробки. IttM — local-first утилита, а не
  SaaS под 10 RPS. Проверять: `maxWorkers: 1` остаётся осмысленным для
  loopback-нагрузки; не вводить очередь воркеров без изменения продукта.
- **Plain HTTP во внутренней Compose-сети.** nginx↔gateway↔OCR не шифруется.
  Это допустимо, потому что трафик замкнут в Docker-сети/loopback. Проверять:
  только nginx опубликован наружу; не включать mTLS без реальной threat-модели.
- **OAuth / Google Drive / DOM-парсинг корзин.** В истории проекта (gemini
  RFC, итерации 1–4) эти сценарии появлялись и отменялись из-за PII/юридических
  рисков. Целевая архитектура работает только с пикселями (скриншоты/PDF) и
  явным LLM-согласием. Проверять: DOM-extractor не возвращается в roadmap;
  не добавлять скрытый сбор PII.
- **`error` event в HTTP 200 после начала NDJSON.** Ошибка посреди стрима
  отдаётся как SSE/NDJSON `error` событие, а не отдельный HTTP status. Это
  компромисс потокового протокола. Проверять: клиент корректно завершает чтение;
  не заменять на replay-протокол без переделки контракта.

## Как использовать этот файл

- При релизе: пройти по **to-do**, подтвердить, что по каждому есть либо
  реализация, либо явное «не делаем в этом релизе» со ссылкой на этот файл.
- При ревью PR: если изменение касается пункта **not-to-do**, убедиться, что
  автор не «починил» компромисс без обновления threat-модели.
